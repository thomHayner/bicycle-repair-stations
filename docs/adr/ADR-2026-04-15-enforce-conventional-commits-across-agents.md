## Context

Most commits in this repo already follow Conventional Commits (`feat(i18n):`, `fix(menu):`, etc.) but the convention is documented only in a four-line section of `CONTRIBUTING.md` and not enforced anywhere. Drift slips in (`Update e2e/share.spec.ts`, `Update public/locales/hy/reportBug.json`).

The project has multiple commit/PR authoring surfaces, each of which needs to behave consistently:

- Humans on the terminal or in an IDE
- VSCode's "Generate commit message with Copilot" sparkle button
- VSCode's "Generate PR title/description with Copilot" button
- Claude Code (this CLI) and its built-in commit workflow
- The GitHub Copilot coding agent / Copilot Workspace
- Other LLM agents (Cursor, Codex, Aider, OpenAI agents)
- Renovate / Dependabot autogenerated PRs
- The GitHub web UI (file edits, merge buttons)

Without explicit guidance plus enforcement, each surface drifts independently and the noise compounds in `git log`.

## Decision

Adopt the Conventional Commits 1.0.0 spec as the single, authoritative format for **all commits and all PR titles**, with one canonical spec in `CONTRIBUTING.md` and thin pointers from every other surface. Enforce with three layers:

1. **Local** — `husky` `commit-msg` hook runs `commitlint` against the in-progress message; bad messages are rejected before the commit completes.
2. **CI on push/PR** — `wagoid/commitlint-github-action@v6` re-validates every commit in a PR.
3. **CI on PR title** — `amannn/action-semantic-pull-request@v5` validates the PR title (catches GitHub web UI / merge-button cases that bypass local hooks).

`commitlint.config.js` extends `@commitlint/config-conventional` with:
- `type-enum` set to severity `2` (error) with a closed allowlist.
- `scope-enum` set to severity `1` (warning) with a curated list — undocumented scopes warn but don't block. This accommodates multi-scope (`i18n+e2e`) and lets the list grow organically.
- Subject case and trailing-period rules at error.
- Header length capped at 72.

Agent-instruction files are kept in lockstep:
- `CLAUDE.md` and `AGENTS.md` carry a short pointer to `CONTRIBUTING.md#commit-messages`.
- `.github/copilot-instructions.md` gains an authoring section above its existing reviewer section.
- VSCode-specific files: `.github/copilot-commit-message-instructions.md` and `.github/copilot-pull-request-description-instructions.md`, opted-in via `.vscode/settings.json`.
- `.claude/commands/commit.md` provides a `/commit` slash command in Claude Code that mirrors the spec.
- `.github/dependabot.yml` configures Dependabot to use `chore`/`ci` prefixes so its autogenerated PRs pass.

## Alternatives considered

- **Document only, no enforcement** — rejected. The convention has been documented since the early days of the repo and drift still happens. Without a hard gate the convention erodes whenever someone is in a rush.
- **PR-title enforcement only, allow noisy commits** — rejected by user choice. Strictest mode (every commit + PR title) was selected; the local hook makes the cost negligible and `git log` stays clean even on branches that aren't squash-merged.
- **Strict closed scope allowlist** — rejected. Forces every new scope to land via a config edit and creates friction for one-off scopes. Warn-level matches how the repo actually evolves.
- **`commitizen` (`git cz` interactive prompt)** — not adopted. Adds an interactive layer that helps humans but doesn't apply to AI agents. The husky/commitlint pair covers the enforcement need without the prompt overhead.
- **Single `AGENTS.md` only, drop `CLAUDE.md` / `copilot-instructions.md`** — rejected. Different agents prioritize different files; keeping all three (with `CONTRIBUTING.md` as the single source of truth) maximizes the chance an agent reads the right thing.

## Consequences

- **Positive**:
  - `git log` becomes machine-parseable, enabling future changelog automation (`conventional-changelog`, semantic-release, etc.) without a separate format-cleanup pass.
  - Bad commits are caught at author time, not at code review.
  - Every authoring surface (human, Claude, Copilot, Cursor, Dependabot) produces consistent output without bespoke prompting per session.
  - PR titles are guaranteed conformant on `main` after squash-merge, including those edited via the GitHub web UI.

- **Negative**:
  - Adds three devDependencies (`@commitlint/cli`, `@commitlint/config-conventional`, `husky`) and ~66 transitive packages.
  - Local commits incur a small overhead (~50–100ms) for the `commit-msg` hook.
  - New contributors hit the lint on their first commit. The `CONTRIBUTING.md` spec is the recovery path; the hook output also points at the violated rule.
  - Dependabot is now enabled where it wasn't before — small ongoing PR volume.

- **Follow-ups**:
  - Historical non-conventional commits (`Update e2e/share.spec.ts`, etc.) are left as-is; rewriting history isn't worth the force-push.
  - Consider adding `conventional-changelog` or `semantic-release` once the convention is settled — out of scope here.
  - Set the repo to "Squash and merge only" in GitHub settings to guarantee the PR title becomes the commit on `main`. (Repo setting, not a file change.)

## Notes

- The `pull_request_template.md` checklists (a11y, dark mode, i18n, MD3, data flow, responsive, docs) are intentionally preserved. AI agents are instructed to **fill in** the template, not replace it; checkboxes remain unchecked for the human author/reviewer to verify.
- Multi-scope (`fix(i18n+e2e):`) works because `scope-enum` is warn-level. If we ever bump it to error, we'd need either a custom plugin or a regex-based scope rule.
- The `husky init` command creates a default `pre-commit` hook running `npm test`. We removed it — pre-commit testing is too slow for this repo's current setup.
