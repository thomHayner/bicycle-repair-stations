## Context

This repo accumulates knowledge across ADRs, merged PRs, technical
specifications, and product docs. Retrieving that knowledge via
ad-hoc RAG over raw files rediscovers structure on every query and
never compounds. We want a persistent synthesis layer that grows over
time.

## Decision

Adopt the Karpathy LLM-Wiki pattern under `wiki/`. `docs/`,
`docs/adr/`, and git history are the raw source layers; `wiki/` is an
LLM-owned synthesis layer with `index.md` as a content catalog and
`log.md` as a chronological record. The operating schema lives in
`CLAUDE.md` so future sessions ingest, query, and lint the wiki
consistently.

## Alternatives considered

- **Ad-hoc RAG over the raw repo** — rejected: re-derives knowledge
  on every query, no accumulated cross-references, no persisted
  synthesis.
- **NotebookLM / ChatGPT file uploads** — rejected: externalizes the
  knowledge base, so it isn't versioned with the code or reviewable
  as part of a PR.
- **Hand-authored wiki (Confluence, GitHub wiki, docs/ alone)** —
  rejected: maintenance cost scales faster than value; humans abandon
  wikis when the bookkeeping gets tedious.
- **Pre-categorized wiki subfolders** (`wiki/entities/`,
  `wiki/concepts/`, etc.) — rejected for now: forces premature
  taxonomy. Flat `pages/` plus an index is simpler and emergent.

## Consequences

- **Positive:** compounding knowledge base; synthesis persists across
  sessions; wiki pages can be cited and audited; cross-reference
  maintenance is LLM work, not human work.
- **Negative:** requires discipline to keep `wiki/` LLM-owned
  (hand-edits must be logged); initial ingest cost for existing
  sources; the schema in `CLAUDE.md` adds upfront instruction weight.
- **Follow-ups:** first ingest pass over existing ADRs, recent PRs,
  and docs; consider local search tooling (e.g., qmd) once page count
  grows past what the index can scan comfortably.
